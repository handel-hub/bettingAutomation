# Audit

## Domain
1.1 Browser Registry State Consistency Audit

## Scope
- `BrowserStateRegistry.mjs`
- `BrowserStateModel.mjs`
- `BrowserCapabilities.mjs`

## Architectural Intent
The `BrowserStateRegistry` is intended to be the authoritative owner and mutator of browser states. It must maintain referential integrity under concurrent mutation and ensure that all state changes are broadcast via the `StateUpdated` event. The `BrowserStateModel` is meant to be a pure data container that is "never mutated by itself".

## Files Examined
- `src/browser/synchronization/BrowserStateRegistry.mjs`
- `src/browser/synchronization/models/BrowserStateModel.mjs`
- `src/browser/synchronization/models/BrowserCapabilities.mjs`

## Execution Model
The registry maintains a `Map` of `BrowserStateModel` instances. Callers retrieve the state via `getState(browserId)`. Mutations are requested via `update(browserId, updates)`, which uses `Object.assign()` to perform partial, in-place updates on the nested context objects (e.g., `navigationContext`, `executionContext`). After mutating the state in-place, it emits a `StateUpdated` event. Since Node.js is single-threaded, the `update()` function executes atomically without preemption.

## Findings

### Finding 1
**Severity**: High
**Category**: State Corruption / Architectural Consistency
**Location**: `BrowserStateRegistry.mjs` -> `getState()`
**Description**: `getState()` returns a direct, mutable reference to the internal `BrowserStateModel` instance. Callers can directly modify properties (e.g., `state.lifecycleState = 'READY'`) or call methods like `state.capabilities.setSatisfied()`, completely bypassing the `update()` method and failing to trigger the `StateUpdated` event.
**Evidence**: 
- Code E1: `getState()` simply returns `this.states.get(browserId)`.
- Code E1: `BrowserStateModel` contains mutable objects and a class instance (`BrowserCapabilities`), meaning direct mutation is syntactically permitted and unchecked.
- Static Analysis E4: Structural inspection confirms there is no `Proxy`, deep copy, or `Object.freeze()` applied to the instance returned by `getState()`. Therefore, any downstream component receiving the object possesses the exact memory address of the internal state map value, establishing an unbounded mutation vector.
**Impact**: If any component accidentally mutates the state directly, critical observers (like `ExecutionScheduler` or `HealthMonitor`) will miss the transition. This will lead to silent desynchronization and orchestrator hangs.
**Root Cause**: Lack of encapsulation. The registry fails to protect its internal state from external mutation.
**Recommendation**: `getState()` should return a read-only projection, deep clone, or a proxy of the state object. Alternatively, enforce strict immutability.
**Confidence**: Very High

### Finding 2
**Severity**: Medium
**Category**: Concurrency
**Location**: `BrowserStateRegistry.mjs` -> `update()`
**Description**: Because the state object is mutated in-place using `Object.assign`, any asynchronous component holding a reference to a nested object (e.g., `state.navigationContext`) across an `await` boundary may see its properties change unexpectedly. This causes dirty reads.
**Evidence**:
- Code E1: `update()` mutates nested objects directly: `Object.assign(state.navigationContext, updates.navigationContext)`.
**Impact**: Logical race conditions. A component might read `state.navigationContext.navigationId`, yield the event loop, and resume execution assuming the context is unchanged, whereas an intervening `update()` call could have mutated it.
**Root Cause**: In-place mutation of nested state objects rather than immutable state replacement.
**Recommendation**: Adopt immutable state transitions. Instead of `Object.assign()`, the registry should replace the entire nested object or the entire state object, allowing components to hold a stable snapshot of the state at the time they read it.
**Confidence**: High

### Finding 3
**Severity**: Critical
**Category**: Performance / Resource Management
**Location**: `BrowserStateRegistry.mjs` -> Memory Lifecycle
**Description**: The registry `Map` grows unbounded over long automation sessions. There is no `removeState`, `delete`, or cleanup method implemented for `this.states`. If browser instances are dynamically spawned and terminated (e.g., via recovery mechanisms or proxy rotations assigning new `browserId`s), old states are permanently retained.
**Evidence**:
- Code E1: `BrowserStateRegistryImpl` implements only `constructor`, `getState`, and `update`.
- Static Analysis E4: A search across the class methods confirms the absence of `this.states.delete()`. Since the map holds strong references to `BrowserStateModel` instances, V8 garbage collection cannot free the memory.
**Impact**: The Node.js isolate will eventually crash with a heap Out-Of-Memory (OOM) error during long-running orchestrator sessions involving many slave rotations.
**Root Cause**: Omission of lifecycle teardown integration in the state registry.
**Recommendation**: Implement a `removeState(browserId)` method and hook it into the `BrowserLifecycleManager`'s teardown events.
**Confidence**: Very High

### Finding 4
**Severity**: High
**Category**: Failure Handling / Consistency
**Location**: `BrowserCapabilities.mjs` -> `setSatisfied()`
**Description**: If `update()` is called with an unregistered capability, `setSatisfied()` silently drops the data instead of throwing an error or registering the capability dynamically.
**Evidence**:
- Code E1: The logic explicitly uses an `if (this.states.has(capability))` guard before mutating the map.
- Execution Trace E2 (Mental Execution Model): A new `CapabilityProvider` emits an update for "NEW_CAP". `BrowserStateRegistry.update` iterates `updates.capabilities` and calls `setSatisfied("NEW_CAP", true)`. The `if` condition fails. The method returns silently. The barrier waiting for "NEW_CAP" times out and crashes the automation.
**Impact**: Developers extending the capability system who forget to add the new capability to the underlying enum will experience indefinite barrier deadlocks that are nearly impossible to trace, as the provider emits correctly but the state silently swallows the signal.
**Root Cause**: Fail-silent error handling for invalid schema inputs.
**Recommendation**: Remove the silent bypass. If `this.states.has(capability)` is false, explicitly `throw new Error('Unknown Capability')` to fail-fast.
**Confidence**: Very High

## Positive Observations
- The `update()` method is synchronous, guaranteeing atomic execution within the Node.js event loop and preventing interleaving updates.
- Deep merging logic for capabilities is correctly segregated to use `setSatisfied` rather than primitive merging.

## Risks
- **RISK-0001**: Leaked mutable state references allowing event-bus bypass.
- **RISK-0002**: Dirty reads of in-place mutated state across asynchronous boundaries.
- **RISK-0003**: Unbounded memory leak of disconnected browser states.
- **RISK-0004**: Silent swallowing of unregistered capability updates causing deadlocks.

## Architectural Weaknesses
The architecture relies on the "honor system" for state immutability. Comments state "It is never mutated by itself. Only the BrowserStateRegistry updates it.", but the language and design do not enforce this, which is a major weakness in a large distributed system.

## Production Readiness
The subsystem is NOT ready for production in its current form. The risks of accidental direct mutation, memory exhaustion over time, and silent synchronization drops make it unsuitable for highly available environments.

## Overall Assessment
The registry correctly serializes updates synchronously but completely fails to protect the integrity of its data structure from external callers, fails to manage resource cleanup, and relies on silent failure for schema invalidity.

## Recommendations
1. Re-architect `BrowserStateModel` to use `Object.freeze()` recursively before returning from `getState()`, or use a Deep Clone strategy.
2. Modify `update()` to construct a new `BrowserStateModel` instance based on the previous state and the partial updates, emitting the new immutable instance. This eliminates dirty reads and enforces the unidirectional data flow.
3. Implement `removeState()` to prevent memory leaks during context teardown.
4. Convert fail-silent capability guards into fail-fast `throw new Error()` assertions.

## Remaining Unknowns
- How many existing subsystems currently rely on holding a direct reference to the state and mutating it directly? If any do, enforcing immutability will require refactoring those components.

## Conclusion
The `BrowserStateRegistry` suffers from significant encapsulation flaws, memory leaks, and error suppression that undermine the event-driven architecture. Enforcing immutability, proper lifecycle memory management, and fail-fast validation are required to satisfy correctness and robustness.
