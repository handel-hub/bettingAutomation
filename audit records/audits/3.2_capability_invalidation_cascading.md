# Audit

## Domain
3.2 Capability Invalidation Cascading Audit

## Scope
- `CapabilityDependencyGraph.mjs`
- `SynchronizationCoordinator.mjs`
- `SynchronizationManager.mjs`
- `capabilities.mjs`

## Architectural Intent
The system ensures that when a core capability (e.g. the browser connection) is lost, all capabilities logically dependent on it (e.g. DOM readiness, Viewport, Scroll) are instantly invalidated. This prevents the macro engine from executing actions against a stale or dead state.

## Findings

### Finding 1
**Severity**: High
**Category**: Missing Root Cascading (Orphaned Capabilities)
**Location**: `CapabilityDependencyGraph.mjs` (Lines 4-25)
**Description**: The statically defined `DOWNWARD_DEPENDENCIES` graph entirely omits the foundational root capabilities `Capabilities.CONNECTED` and `Capabilities.SESSION_READY` (which are defined in `capabilities.mjs`). Because they are missing from the dependency graph, `CapabilityDependencyGraph.getDependentCapabilities('CONNECTED')` returns an empty array.
**Evidence**:
- Code E1: `CapabilityDependencyGraph.mjs` lacks `CONNECTED`.
- System Context E2: The `SynchronizationCoordinator._invalidateDependencies` relies purely on `getDependentCapabilities`. If `CONNECTED` drops, no dependent capabilities (like `NAVIGATION_READY` or `DOM_READY`) are yielded.
- Static Analysis E4: If the underlying WebSocket drops (proxy failure), `ConnectionCapability` will transition to `false`. Because `DOM_READY` remains `true` in `capabilityStates`, Playwright will attempt to execute locator queries against a disconnected page, throwing fatal errors.
**Impact**: Fatal execution errors when network partitions or proxy drops occur, completely breaking macro recovery because the system doesn't realize the DOM is dead.
**Root Cause**: Incomplete static mapping of the capabilities graph.
**Recommendation**: Add `Capabilities.CONNECTED` to `DOWNWARD_DEPENDENCIES` and `UPWARD_DEPENDENCIES`, mapping it to all child capabilities (Navigation, DOM, Viewport, Scroll, Frame).

### Finding 2
**Severity**: High
**Category**: Context-Bound Invalidation Failure (Stale Providers)
**Location**: `SynchronizationManager.mjs` (Lines 20-28)
**Description**: The `SynchronizationManager` listens to the coordinator's `InvalidationRequested` event and passes it to the corresponding Provider so it can clean up its internal state (via `provider.invalidate(context)`). However, it retrieves the context via `this.activeSyncContexts.get(event.browserId)`. This map is *only* populated during an active `awaitCapabilities` tick. If an out-of-band invalidation occurs (e.g., a spontaneous CDP event drops a connection while the browser is idle), the `context` is undefined, and the provider is NEVER notified of the invalidation.
**Evidence**:
- Code E1: `SynchronizationManager.mjs` Lines 51 and 57 (`this.activeSyncContexts.set` and `.delete`) bound the context strictly to the duration of `awaitCapabilities`.
- Static Analysis E4: The provider's internal state becomes stale. If it maintained timers, listeners, or cached resolutions, they will not be cleared.
**Impact**: Stale capability utilization and memory leaks from orphaned provider listeners.
**Root Cause**: Assuming that invalidations only occur when the system is actively waiting for capabilities.
**Recommendation**: Providers must not strictly require a `syncContext` to invalidate internal cache state. `SynchronizationManager` should always call `provider.invalidate(event.browserId)` even if there is no active `syncContext` available.

## Positive Observations
- **Protection Against Circular Invalidation**: The invalidation engine completely bypasses circular dependency loops. `_invalidateDependencies()` explicitly loops over a statically defined array and manually flags capabilities to `false` without recursive function calls, preventing call-stack overflows perfectly.

## Overall Assessment
The cascading invalidation system works cleanly for DOM and layout capabilities, but utterly fails at the most critical level: the root connection. Furthermore, it only informs Providers of invalidation if it happens while actively executing a sync barrier.

## Recommendations
1. Integrate `CONNECTED` and `SESSION_READY` into `CapabilityDependencyGraph`.
2. Refactor `SynchronizationManager` so that `provider.invalidate()` can accept a raw `browserId` instead of a full `syncContext` to ensure background invalidations succeed.

## Conclusion
Resolving the root capability mapping and decoupling invalidation from active sync cycles is required for true autonomous system recovery.
