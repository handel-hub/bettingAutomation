# Audit

## Domain
5.1 Locator Generation Main-Thread Latency Audit

## Scope
- `ActionDispatcher.mjs` (`InteractionRecognizer.emit()`)
- `LocatorIntelligenceEngine.mjs`
- `FeatureExtractor.mjs`
- `StructuralStrategy.mjs`

## Architectural Intent
The locator generation pipeline must capture an immutable snapshot of the DOM and generate a resilient, cross-session locator whenever a human interacts with an element. Because modern web apps (React, Vue) mutate the DOM instantaneously upon interaction, this capture process must happen synchronously within the event listener before the element is detached.

## Findings

### Finding 1
**Severity**: High
**Category**: Synchronous Layout Thrashing (Reflow)
**Location**: `FeatureExtractor.mjs` (Line 40)
**Description**: The `FeatureExtractor` attempts to read the visible text of the target element's children using `node.innerText || node.textContent`. 
**Evidence**:
- Code E1: `textContent += node.innerText || node.textContent || '';`
- Static Analysis E4: `node.innerText` is an incredibly expensive DOM API because it is style-aware. Evaluating it forces the browser's rendering engine to flush its pending layout queue and perform a synchronous reflow on the main thread. If a user clicks an element while the page layout is "dirty" (e.g., an animation is running or CSS classes just changed), this single property access can block the main thread for 50-200ms, causing massive UI jitter for the human operator.
**Impact**: The master browser will feel laggy, unresponsive, and janky during recording sessions.
**Recommendation**: Remove `node.innerText` entirely. Rely exclusively on `node.textContent` (which operates purely on the DOM AST and never triggers reflows) and perform post-processing (e.g., regex whitespace normalization) in JavaScript.

### Finding 2
**Severity**: High
**Category**: Unbounded O(N) Sibling Traversal
**Location**: `StructuralStrategy.mjs` (Lines 26-29)
**Description**: To calculate the `:nth-of-type` selector, the strategy uses a `while (sib = sib.previousElementSibling)` loop for the element and up to 10 of its parent nodes.
**Evidence**: 
- Code E1: 
```javascript
let sib = current, nth = 1;
while (sib = sib.previousElementSibling) {
    if (sib.nodeName.toLowerCase() == selector) nth++;
}
```
- Static Analysis E4: If the user clicks on the 5,000th `<tr>` in a massive data table, this loop will synchronously iterate 5,000 times. If the table is nested, it may happen multiple times.
**Impact**: Severe CPU spikes on the main thread for specific DOM topologies (long lists, grids, tables).
**Recommendation**: Enforce a strict upper bound on sibling traversal (e.g., `if (nth > 50) break;`). If an element is the 51st sibling, an `nth-of-type(51)` selector is inherently brittle anyway, so abandoning the structural strategy early is both safer and significantly faster.

### Finding 3
**Severity**: None (Architectural Validation)
**Category**: Offloading Viability
**Description**: The blueprint asks: "Can generation be offloaded to `requestIdleCallback` or Web Workers?"
**Analysis**: 
- **Web Workers**: Impossible. Workers do not have access to the DOM, and serializing the `Element` (with its parent/sibling references) is prohibited by V8.
- **requestIdleCallback**: Highly risky. Modern SPAs often destroy or detach elements immediately upon a `click` event. If locator generation is deferred to an idle frame, the `el.parentNode` traversal will return `null` because the element is already orphaned, resulting in a total failure to generate a locator.
**Conclusion**: The generation *must* remain synchronous. The only valid solution is micro-optimizing the pipeline (as recommended in Findings 1 and 2) to execute in <5ms.

## Overall Assessment
The locator pipeline correctly anchors itself synchronously to prevent detached-node errors, but its internal implementation utilizes APIs (`innerText`) and unbounded loops (`previousElementSibling`) that violate strict 60fps main-thread performance budgets.

## Recommendations
1. Purge all references to `innerText` or `getBoundingClientRect()` that aren't strictly guarded or deferred. Use `textContent`.
2. Cap sibling traversal at a low threshold (e.g., 20-50).

## Conclusion
Resolving the forced reflows and unbounded loops will bring the P99 execution time well under 10ms, eliminating UI jitter without compromising locator accuracy.
