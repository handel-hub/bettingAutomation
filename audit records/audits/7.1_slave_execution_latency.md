# Audit

## Domain
7.1 Slave Playwright Execution Engine Latency Audit

## Scope
- `ActionSimulator.mjs` (Action Execution Layer)
- `LocatorResolver.mjs` (Resolution Loop)
- `ExecutionScheduler.mjs` (Queue Drain Mechanism)

## Architectural Intent
The Slave execution engine must rapidly mirror the interactions recorded by the Master. In betting contexts, a bet sequence must be executed within tight time windows (often < 5 seconds). The Slave must execute generated locators with minimal latency to avoid falling behind the live data stream.

## Findings

### Finding 1
**Severity**: High
**Category**: Unbounded Auto-Waiting Execution Hangs
**Location**: `ActionSimulator.mjs` (Line 86, 107, 117)
**Description**: The framework attempts to implement a fast-path resolution loop by wrapping `LocatorResolver` in a 2,500ms global timeout. However, the subsequent execution phase invokes Playwright's API (`await loc.click()`, `await loc.fill()`) without explicit timeout parameters.
**Evidence**:
- Code E1: 
```javascript
usedLocatorInfo = await this._executeWithRecovery(command, page, 'click', async (loc) => await loc.click());
```
- Code E2: In `ActionSimulator.mjs`, `MAX_EXECUTION_RETRIES` is set to 3.
- Playwright's default action timeout is 30,000ms. If a target element is physically obscured by a transient overlay (e.g., a cookie banner or a loader spinner), Playwright's built-in actionability checks will stall the `click()` call for 30 seconds.
- Because `ActionSimulator` catches `TimeoutError` and retries 3 times, a single intercepted click will stall the entire `async` execution promise for **90 seconds**.
**Impact**: The betting window entirely expires. The automation process falls catastrophically behind real-time.

### Finding 2
**Severity**: High
**Category**: Sequential Execution Queue Blocking
**Location**: `ExecutionScheduler.mjs` (Line 288)
**Description**: The queue drain mechanism processes commands sequentially per browser ID.
**Evidence**:
- Code E1:
```javascript
while (true) {
    const nextEntry = qManager.dequeueNext();
    // ...
    await this.simulator.execute(browserObj, finalCommand);
}
```
- When `simulator.execute` stalls for 90 seconds due to Finding 1, the `ExecutionScheduler` is blocked on the `await`. All subsequent commands in the queue for that browser are frozen.
**Impact**: A single overlay delay on a non-critical command cascades, artificially delaying time-critical commands (like submitting the betslip) queued behind it.

### Finding 3
**Severity**: Medium
**Category**: Redundant Actionability Validation Overhead
**Location**: `LocatorResolver.mjs` (Line 97) & `ActionSimulator.mjs` (Line 86)
**Description**: The `LocatorResolver` manually queries the browser via CDP to assert visibility and actionability.
**Evidence**:
- Code E1 (`LocatorResolver.mjs`):
```javascript
if (!(await locator.isVisible())) throw new HiddenError('Visible: No');
if (!(await locator.isEnabled())) throw new DisabledError('Enabled: No');
```
- Code E1 (`ActionSimulator.mjs`):
```javascript
await loc.click();
```
- Playwright's `click()` inherently re-verifies attachment, visibility, stability, and enablement via its own internal CDP calls.
**Impact**: The Slave incurs a double-penalty in network round-trips for every interaction. In high-speed bulk execution (e.g., executing 50 DOM interactions), these redundant validations sum up to hundreds of milliseconds of wasted latency.

## Overall Assessment
The Slave execution engine contradicts the architectural goals of mirroring by implicitly trusting Playwright's auto-wait mechanisms. By failing to supply strict fail-fast timeout boundaries to physical execution commands, the framework is vulnerable to massive latency spikes (up to 90 seconds) during overlay interceptions.

## Recommendations
1. **Pass Explicit Action Timeouts**: Force all Playwright physical executions to fail fast by passing explicit timeouts matching the framework's internal SLA (e.g., `await loc.click({ timeout: 1000 })`).
2. **Utilize Force Mode**: Since `LocatorResolver` already pre-validates visibility and enablement, consider appending `{ force: true }` to `click()` calls to bypass Playwright's redundant actionability and stability checks, vastly accelerating execution.
3. **Fail Fast Strategy**: In mirroring automation, if an element is intercepted, waiting 30 seconds for an overlay to clear is fatal. The Slave should fail instantly and trigger a hard page reload or recovery sequence rather than attempting to auto-wait.

## Conclusion
The Slave's physical execution layer is dangerously slow due to inherited default timeouts. Without strict millisecond bounds on `actionFn` invocations, the master-slave synchronization will permanently desynchronize during dynamic UI states.
