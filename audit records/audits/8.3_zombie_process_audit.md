# Audit

## Domain
8.3 Context Teardown & Zombie Process Audit

## Scope
- `RecoveryManager.mjs` (`closeQuietly`)
- `BrowserLifecycleManager.mjs` (Process initialization)

## Architectural Intent
A distributed browser automation framework operating hundreds of proxies will inevitably experience broken sockets, proxy timeouts, and Chromium hangs. When a slave browser becomes unresponsive and is targeted for recovery, the framework must guarantee that the underlying OS process is destroyed and its temporary disk artifacts are wiped, regardless of the browser's responsiveness to graceful shutdown commands.

## Files Examined
- `src/browser/coordination/RecoveryManager.mjs`
- `src/browser/coordination/BrowserLifecycleManager.mjs`

## Findings

### Finding 1
**Severity**: High
**Category**: Indefinite Process Hang (Zombie Leak)
**Location**: `RecoveryManager.mjs` (Lines 74-80, `closeQuietly`)
**Description**: The recovery logic attempts to tear down a stale browser using `await browserObj.browser?.close()`. Playwright's `.close()` method relies on the Chrome DevTools Protocol (CDP) WebSocket. If the reason the browser was flagged for recovery is a deadlocked or frozen renderer/CDP socket, the `.close()` promise will hang indefinitely. Because the framework does not wrap this in a timeout or fall back to an OS-level `kill()`, the `heal()` function is blocked forever, and the headless Chromium process becomes an immortal zombie.
**Evidence**:
- Code E1: `closeQuietly` awaits `.close()` natively without a `Promise.race` timeout wrapper.
- Static Analysis E4: If `browser.close()` hangs, execution never proceeds to `registry.remove()` or `respawn()`, halting that slave's automation track permanently and leaking ~200MB of RAM per incident.
**Impact**: Gradual server RAM exhaustion, PID table exhaustion, and permanent loss of automation capacity as slaves get permanently stuck in the `heal` phase.
**Root Cause**: Assuming a graceful CDP shutdown is possible on a system that is known to be in an unhealthy state.
**Recommendation**: Implement a strict timeout (e.g., 5000ms) on `browser.close()`. If it times out, retrieve the OS process ID via `browser.process().pid` and execute a forceful `process.kill(pid, 'SIGKILL')`.
**Confidence**: Absolute

### Finding 2
**Severity**: Medium
**Category**: Disk/Inode Exhaustion via Orphaned Profiles
**Location**: `RecoveryManager.mjs` / `BrowserLifecycleManager.mjs`
**Description**: When Playwright launches a non-persistent context (via `chromium.launch`), it creates a temporary user data directory (usually in `/tmp` or `%TEMP%`). Playwright's internal cleanup hooks only delete this directory if `browser.close()` completes successfully or the Node.js process exits gracefully. Due to Finding 1 (where `.close()` hangs or fails), these temporary folders are abandoned.
**Evidence**:
- Code E1: There is no manual tracking or rimraf cleanup of the `userDataDir` on recovery failure.
- Static Analysis E4: Over a 24-hour cycle with thousands of proxy rotations and crashes, orphaned temporary profiles will silently exhaust the host system's disk space and inodes.
**Impact**: Eventual `ENOSPC` (No space left on device) errors crashing the entire host server.
**Root Cause**: Implicit reliance on upstream library cleanup hooks during catastrophic failure scenarios.
**Recommendation**: Explicitly pass a custom `userDataDir` path (e.g., `/tmp/baaos_slave_<id>`) during `chromium.launch`. During `closeQuietly()`, manually execute `fs.rmSync(path, { recursive: true, force: true })` after the process is verified dead.
**Confidence**: High

## Positive Observations
- The `RecoveryManager` utilizes a `finally` block to delete `healingIds`, ensuring the lock is released if an error occurs.
- The system correctly attempts to remove the browser from the `registry` during teardown (though this is blocked by Finding 1).

## Overall Assessment
The teardown sequence is "happy-path" oriented and completely unsafe for production execution at scale. It trusts a broken browser to gracefully shut itself down.

## Recommendations
1. Wrap all `.close()` calls in a `Promise.race` timeout.
2. Track and `SIGKILL` the Chromium OS process ID upon timeout.
3. Manage and forcefully purge custom `userDataDir` paths.

## Conclusion
The teardown logic suffers from severe resource leaks (RAM, CPU, and Disk). Immediate defensive programming is required to ensure OS-level process termination.
