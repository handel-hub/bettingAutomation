# Audit

## Domain
4.4 Queue Priority Inversion & Starvation Audit

## Scope
- `ExecutionScheduler.mjs` (`QueueManager.dequeueNext()`, `ClassificationPolicy`)

## Architectural Intent
The queueing system attempts to prioritize "important" discrete events (Clicks, Inputs, Navigations) over "noisy" continuous events (Hovers, Scrolls) to ensure that the system remains highly responsive to core business logic, preventing the Playwright execution engine from getting bogged down simulating thousands of mouse coordinate movements.

## Findings

### Finding 1
**Severity**: Critical
**Category**: Temporal Priority Inversion (Chronological Desync)
**Location**: `ExecutionScheduler.mjs` (Line 110-128)
**Description**: Because `dequeueNext()` strictly evaluates the `buckets` array in a fixed order (`Critical -> Discrete -> Aggregated -> Continuous`), it completely destroys the temporal chronology of user interactions if the queue backs up. If a slave browser is temporarily blocked by a synchronization barrier for 500ms, a user's rapid sequence of `Scroll` -> `Hover` -> `Click` will all arrive in the queue simultaneously. The scheduler will reorder them to execute the `Discrete` (Click) first, the `Aggregated` (Scroll) second, and the `Continuous` (Hover) last.
**Evidence**: 
- Code E1: `const order = ['Critical', 'Discrete', 'Aggregated', 'Continuous'];` guarantees `Discrete` is drained before `Aggregated`.
- Static Analysis E4: If a `CLICK` targets an element inside a dynamic dropdown menu that requires a `HOVER` to become visible, the reordered execution will attempt the `CLICK` while the dropdown is closed, causing Playwright to throw a timeout/visibility error.
**Impact**: Complex, multi-step UI interactions (like navigating mega-menus or lazy-loaded scroll grids) will fail deterministically on slaves if there is even minor network latency that allows events to bunch up in the queue.
**Root Cause**: Fundamentally flawed assumption that event "types" supersede chronological causality in stateful DOM environments.
**Recommendation**: Replace the strict class-based priority queues with a single chronological queue. For performance, implement "Look-ahead Coalescing"—scan the single chronological queue and prune/coalesce contiguous hover/scroll events, but never reorder them relative to `Discrete` events (e.g., never move a hover past a click).

### Finding 2
**Severity**: High
**Category**: Queue Starvation
**Location**: `ExecutionScheduler.mjs` (Line 114)
**Description**: The `while (bucket.length > 0)` loop for `Discrete` commands must completely exhaust the `Discrete` bucket before `dequeueNext()` ever considers the `Aggregated` or `Continuous` buckets. 
**Evidence**:
- Code E1: The outer `for (const queueClass of order)` loop ensures absolute starvation of lower tiers.
- Static Analysis E4: If a user types a long string of text quickly (generating 10 `INPUT`/`KEYBOARD` events per second), the time required for Playwright to execute these `Discrete` events will continuously overlap with the arrival of new `Discrete` events. The `Discrete` bucket will never fully empty, meaning any simultaneous mouse movements (`Continuous`) will sit untouched until they trigger the 1500ms TTL and are deleted.
**Impact**: Slave browsers will freeze their viewports and cursors entirely while text input or rapid clicking is occurring, making anti-bot detection trivially easy due to robotic, parallelized event isolation (typing without mouse jitters).
**Root Cause**: Strict priority scheduling without fairness or aging mechanisms.
**Recommendation**: If multiple queues are maintained, implement Deficit Round Robin (DRR) or enforce a maximum drain limit (e.g., drain a max of 3 `Discrete` events before forcing a poll of the `Continuous` queue).

## Overall Assessment
The `ExecutionScheduler` applies operating-system level CPU scheduling concepts (Strict Priority Queues) to a stateful UI domain where those concepts are violently destructive. UI interactions are causally linked (A must happen before B); prioritizing B over A mathematically guarantees execution failure.

## Recommendations
1. Abandon the four discrete buckets for standard execution. Use a single chronological FIFO queue to preserve causality.
2. Maintain `Critical` as an interrupt queue (for recovery actions only) that *can* preempt the FIFO queue, as recovery by definition resets the DOM state.
3. Perform coalescing and dropping inline during `enqueue` rather than relying on segregated buckets.

## Conclusion
The prioritization logic currently guarantees macro failure under real-world network latency by destroying the chronological order of dependent DOM interactions.
