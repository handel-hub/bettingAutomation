# Audit

## Domain
8.2 Proxy Failure Graceful Degradation Audit

## Scope
- `ProxyManager.mjs`
- `AutomationController.mjs` (Browser Bootstrapping)
- `RecoveryManager.mjs` (Healing Sequences)

## Architectural Intent
The system must never expose the physical host IP address to the target platform. If a proxy fails at boot, the system should strictly refuse to launch the account. If a proxy fails mid-session, the system should gracefully rotate the proxy and recover the session without permanently abandoning the slave.

## Findings

### Finding 1
**Severity**: High
**Category**: Silent Proxy Bypassing via Loose Mode
**Location**: `AutomationController.mjs` (Lines 270, 288), `ProxyManager.mjs` (Line 14)
**Description**: The default configuration allows browsers to launch completely unprotected if the proxy allocation fails. 
**Evidence**:
- Code E1 (`ProxyManager.mjs`):
```javascript
this.failureMode = this.settings.proxy_failure_mode || 'loose';
```
- Code E2 (`AutomationController.mjs`):
```javascript
const proxyUrl = this.proxyManager.allocateProxy();
if (!proxyUrl && this.settings.Proxy.proxy_failure_mode === 'strict') {
    // Skips account
}
await this.lifecycleManager.spawnBrowser(id, 'slave', proxyUrl, account.username);
```
- Static Analysis E4: If the proxy mode is `loose` (the default) and `proxyUrl` is `null` (because proxies are offline or capacity is exhausted), the system bypasses the `if` block and executes `spawnBrowser(..., null, ...)`. Playwright will then launch the browser instance using the host machine's default network interface.
**Impact**: Multiple accounts will instantly leak the master datacenter IP to the target website, resulting in cascading permanent account bans.

### Finding 2
**Severity**: High
**Category**: Lack of Mid-Session Proxy Rotation
**Location**: `RecoveryManager.mjs` (Line 32, 42)
**Description**: When a residential proxy naturally rotates or drops offline mid-session, the slave correctly crashes (Playwright fails closed). However, the `RecoveryManager` fails to provision a new proxy.
**Evidence**:
- Code E1 (`RecoveryManager.mjs`):
```javascript
const target = this.registry.get(browserId);
const { role, username, proxyUrl } = target;
// ...
await this.respawn(browserId, role, proxyUrl, username);
```
- Static Analysis E4: The `RecoveryManager` extracts the dead `proxyUrl` from the crashed browser's state registry and injects it back into the `respawn` command. It loops this 3 times (`maxAttempts`) before emitting a fatal `HEAL_FAILED` event, permanently destroying the slave. 
**Impact**: Transient proxy failures (which are expected daily with rotating residential proxy pools) cause permanent slave death instead of graceful rotation.

## Overall Assessment
The proxy management system contains two critical architectural flaws. First, it prioritizes "booting successfully" over "booting securely" by defaulting to a loose failure mode that leaks IPs. Second, its recovery layer is perfectly rigid, blindly reusing dead network credentials instead of requesting fresh routes from the `ProxyManager`.

## Recommendations
1. **Enforce Strict Mode by Default**: Change the default `proxy_failure_mode` to `strict` in `ProxyManager.mjs`. The framework should *never* spawn an unprotected browser unless explicitly requested by a developer debugging locally.
2. **Implement Proxy Rotation in Recovery**: Modify `RecoveryManager.heal()` to detect network-related crashes. If a crash is network-bound, it should invoke `proxyManager.allocateProxy()` to fetch a fresh proxy URL before calling `respawn()`, and update the `BrowserStateRegistry` with the new proxy string.

## Conclusion
Playwright behaves correctly (failing closed when proxy bound), but the framework wrapper compromises this safety through loose defaults and static credential reuse, resulting in guaranteed stealth leaks and permanent slave attrition.
