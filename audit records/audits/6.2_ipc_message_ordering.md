# Audit

## Domain
6.2 IPC Message Ordering and Causal Consistency Audit

## Scope
- `ActionDispatcher.mjs` (`exposeBinding` callback, `recordAction`)

## Architectural Intent
The framework must guarantee that interactions performed by the human operator on the master browser are replicated on the slave browsers in the exact same chronological sequence, preserving strict causal reality (e.g., a modal must be clicked *before* the subsequent keystrokes are typed into it).

## Findings

### Finding 1
**Severity**: Critical
**Category**: Causal Desynchronization via IPC Race Conditions
**Location**: `ActionDispatcher.mjs` (Lines 40, 105)
**Description**: The framework processes and records events strictly by their Node.js IPC arrival time, rather than sorting them by their V8 capture timestamp (`captureTime`).
**Evidence**:
- Code E1: 
```javascript
// Inside exposeBinding callback:
this.recordAction(eventData); // Pushes directly to the end of the array
this.emit('Command', command); // Immediately dispatches to the slaves
```
- Static Analysis E4: JavaScript is single-threaded per frame, so events within a single frame are strictly ordered. However, if a page uses cross-origin iframes, those iframes run in parallel V8 isolates. 
If the user clicks a button in Frame A at `T=100ms`, the `LocatorIntelligenceEngine` will block Frame A's thread for ~50ms to calculate the complex locator. If the user then scrolls in Frame B at `T=110ms`, Frame B's lightweight scroll event takes ~1ms to process.
Frame B transmits its payload over the CDP websocket at `T=111ms`.
Frame A transmits its payload over the CDP websocket at `T=150ms`.
The Node.js `ActionDispatcher` receives the SCROLL before the CLICK, blindly records it in `sequence.json` in that invalid order, and instantly emits it to the `ExecutionScheduler`. The slave browser will execute the scroll *before* the click, completely destroying causal reality and breaking the automation sequence.
**Impact**: Multi-frame applications will suffer from impossible-to-debug playback failures where actions occur out of order.
**Root Cause**: The Node.js receiver assumes the network (CDP) arrival order mirrors the chronological user action order, failing to account for payload generation latency differentials across parallel V8 isolates.
**Recommendation**: 
1. Implement a **Jitter Buffer** in the `ActionDispatcher` before emitting commands or writing to `this.actions`. 
2. When an event arrives, hold it in a priority queue sorted by `eventData.payload.timestamp` (or `captureTime`) for a small window (e.g., 100ms).
3. Only release/emit the events from the buffer to the `ExecutionScheduler` once the 100ms window guarantees no older events are still generating in slower frames.

## Overall Assessment
The IPC ingestion pipeline is architecturally naive regarding distributed systems timing. By trusting network arrival order over logical timestamps, it exposes the orchestration to severe race conditions whenever multi-frame parallelism is involved.

## Recommendations
Introduce a mandatory chronological reordering buffer (Jitter Buffer) at the Node.js ingestion boundary.

## Conclusion
The current IPC message processing cannot guarantee causal consistency across parallel iframes. Reordering logic must be implemented immediately to prevent sequence corruption.
