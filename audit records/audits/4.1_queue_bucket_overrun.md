# Audit

## Domain
4.1 Queue Bucket Overrun & Memory Audit

## Scope
- `src/browser/execution/ExecutionScheduler.mjs` (specifically `QueueManager`, `SchedulingPolicy`, `ClassificationPolicy`)

## Architectural Intent
The `ExecutionScheduler` buffers incoming physical interactions into distinct queue buckets (`Critical`, `Discrete`, `Aggregated`, `Continuous`). To ensure system stability against event floods, these queues must employ strict memory bounding, TTL expiration, and state coalescing logic to prevent unbounded memory growth and eventual Node.js V8 heap exhaustion (OOM crashes).

## Files Examined
- `ExecutionScheduler.mjs`

## Execution Model
1. Incoming commands are mapped to a `queueClass` and priority.
2. `QueueManager.insert()` validates bucket lengths.
3. `SchedulingPolicy.apply()` merges or pushes the payload into the respective `queueBucket`.
4. `QueueManager.dequeueNext()` yields commands based on priority and TTL limits.

## Findings

### Finding 1
**Severity**: Critical
**Category**: Unbounded Queue Growth / OOM Vulnerability
**Location**: `ExecutionScheduler.mjs` (Lines 97-100, `QueueManager.insert`)
**Description**: The system implements an explicit safeguard against queue flooding by restricting the `Discrete` bucket to a maximum of 100 items, throwing a fatal error if exceeded. However, this check is entirely absent for the `Critical` bucket. Because `SchedulingPolicy.apply()` pushes `Critical` commands unconditionally, any upstream logic failure that emits rapid `Recovery` or `Navigation` events (such as a routing loop or a rapid retry cascade) will cause the `buckets.Critical` array to grow infinitely.
**Evidence**:
- Code E1: Lines 97-99 explicitly restrict `Discrete`: `if (queueClass === 'Discrete' && this.buckets.Discrete.length >= 100)`. There is no corresponding check for `queueClass === 'Critical'`.
- Execution Trace E2:
  1. `t=0`: A localized network outage drops the Playwright socket for a specific slave.
  2. `t=1`: `HealthMonitor` detects the dead socket and emits `HEAL_REQUESTED`.
  3. `t=2`: `RecoveryManager` creates a `Command` with `category: 'Recovery'` (classified as `Critical`) and calls `ExecutionScheduler.enqueue()`.
  4. `t=3`: `QueueManager.insert()` receives the command. Because `queueClass === 'Critical'`, the boundary check on Line 97 is bypassed, and the command is pushed to the array unconditionally.
  5. `t=4`: The scheduler's `_drain` loop invokes `SynchronizationBarrier.wait()`. Due to the starvation bug found in Domain 3.3, the barrier hangs infinitely. The `drainLock` is never released.
  6. `t=5 to t=10000`: `HealthMonitor` continues polling every 500ms, emitting repeated `HEAL_REQUESTED` events. `RecoveryManager` responds by enqueuing thousands of new `Critical` commands.
  7. `t=Inf`: Because the queue cannot drain and there is no length limit on `buckets.Critical`, the array grows indefinitely until the Node.js V8 Heap exhausts and the Master orchestration process fatally crashes (OOM).
**Impact**: Complete System-wide Denial of Service (OOM Crash). A single failing slave emitting a flood of recovery events will take down the entire distributed orchestration cluster, terminating all healthy workflows.
**Root Cause**: Incomplete application of array boundary checks across all dynamically scaling queue classes.
**Recommendation**: Implement a hard limit for the `Critical` bucket (e.g., 50 items). If exceeded, the system must trigger a catastrophic `KILL_SLAVE` event rather than storing the command, as >50 pending critical recovery events indicates an unrecoverable cascading failure loop.
**Confidence**: Very High

### Finding 2
**Severity**: High
**Category**: Ephemeral Context Memory Leak
**Location**: `ExecutionScheduler.mjs` (Lines 304-308, `ExecutionScheduler.clearQueue`)
**Description**: The `ExecutionScheduler` instantiates a new `QueueManager` for every connecting slave browser, keyed by `browserId` in `this.browserQueues`. When a browser terminates, `clearQueue(browserId)` is available to empty the internal arrays (`buckets`). However, the `ExecutionScheduler` completely lacks a mechanism to invoke `this.browserQueues.delete(browserId)`. 
**Evidence**:
- Code E1: The `browserQueues` Map has `.set()` and `.get()` operations, but `.delete()` is never called anywhere in the codebase.
- Static Analysis E4: Key-value pairs inserted into a standard `Map` establish strong references to the `QueueManager` objects. As ephemeral slaves rotate through proxies or restart, the Map will indefinitely accumulate empty instances.
**Impact**: Gradual unbounded memory leak. Over a 24-hour automation session where tens of thousands of slave contexts are generated and destroyed for evasion rotation, the Node.js process will eventually run out of memory.
**Root Cause**: Misalignment between object clearing (`clear()`) and structural detachment (`Map.delete()`).
**Recommendation**: Introduce a `removeBrowser(browserId)` method in `ExecutionScheduler` that explicitly executes `this.browserQueues.delete(browserId)` and `this.drainLocks.delete(browserId)`. Ensure `BrowserLifecycleManager` invokes this during its teardown phase.
**Confidence**: Very High

## Positive Observations
- The `Continuous` and `Aggregated` buckets are perfectly bounded. The logic enforces a strict maximum array length of 1 by aggressively overwriting (`queueBucket[0] = entry`) or mathematically coalescing payloads.
- `QueueManager.handleNavigation()` successfully implements memory hygiene by instantly flushing stale hover/scroll events upon navigation triggers.
- The use of `deepFreeze` on the `Command` payload guarantees deterministic immutability during queue residence.

## Risks
- **RISK-NEW-17**: Infinite `Critical` queue growth causing OOM crashes.
- **RISK-NEW-18**: Long-term memory leaks in structural Maps leading to gradual performance degradation.

## Architectural Weaknesses
The architecture relies on implicit garbage collection through object clearing rather than explicit lifecycle attachment/detachment of the `browserId` lifecycle bindings.

## Production Readiness
Not ready. The lack of bounds on the `Critical` queue is a fatal stability flaw in distributed systems where cascading failure events are common.

## Overall Assessment
The `QueueManager` successfully solves the Continuous/Aggregated flooding problem but fails the Discrete/Critical boundary enforcement test.

## Recommendations
1. Enforce a hard ceiling on `Critical` queues.
2. Implement explicit `Map.delete()` cleanup hooks for disconnected slaves.

## Conclusion
The Queue bucket logic is resilient to mouse movement flooding but critically vulnerable to recovery event flooding. Immediate memory management corrections are required.
