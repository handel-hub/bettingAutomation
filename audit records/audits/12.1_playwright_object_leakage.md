# Audit

## Domain
12.1 Playwright Object Leakage Audit

## Scope
- `BrowserStateRegistry.mjs`
- `BrowserStateModel.mjs` (and associated Contexts: `NavigationContext`, `ExecutionContext`)
- `BrowserRegistry.mjs` (Legacy)
- Execution Models (`ResolutionResult`, `Command`)
- Ephemeral Trackers (`NavigationTracker`, `FrameTracker`)

## Architectural Intent
The architecture dictates that no Playwright-specific objects (`Page`, `Browser`, `ElementHandle`, `Locator`) are stored in persistent memory layers, registries, or DTOs. These objects must only be passed ephemerally by reference through execution parameters, ensuring the browser framework remains cleanly decoupled from the state synchronization and intelligence pipelines, and avoiding fatal memory leaks or dead-object crashes upon browser disconnections.

## Findings

### Finding 1
**Severity**: High
**Category**: Memory Leak & Framework Lock-in
**Location**: `src/browser/coordination/BrowserRegistry.mjs` (Lines 6-18)
**Description**: The codebase contains a legacy state registry (`BrowserRegistry.mjs`) that actively stores physical Playwright objects (`browser`, `context`, `page`) in a persistent map alongside application state metadata (`state`, `url`, `health`).
**Evidence**:
- Code E1 (`BrowserRegistry.mjs`):
```javascript
    register(id, role, browser, context, page, meta = {}) {
        this.browsers.set(id, {
            id,
            role,      // 'master' | 'slave'
            state: 'Initializing', 
            url: 'about:blank',
            health: 'Good', 
            browser,   // <-- Playwright object stored in state map
            context,   // <-- Playwright object stored in state map
            page,      // <-- Playwright object stored in state map
            username: meta.username ?? null,
            proxyUrl: meta.proxyUrl ?? null
        });
    }
```
- Code E2 (`SessionManager.mjs` Line 122): `const page = browserObj.page;` - Further evidence that execution relies on retrieving the physical page object from this persistent registry.
- Static Analysis E4: Because these objects are stored indefinitely in `this.browsers`, they will never be garbage collected even if the underlying browser crashes, leading to severe memory leaks. Furthermore, accessing a dead `page` object from this registry will crash the Node.js application.

### Finding 2
**Severity**: None (Validation Success)
**Category**: Clean Data Models
**Location**: `src/browser/synchronization/models/BrowserStateModel.mjs`
**Description**: The modern synchronization layer (`BrowserStateRegistry`, `BrowserStateModel`, `NavigationContext`, `ExecutionContext`) correctly implements the architecture. No Playwright objects are leaked into the official state synchronization containers.
**Evidence**:
- Static Analysis E4: An exhaustive search of `BrowserStateModel.mjs` and the `BrowserStateRegistry.update()` payloads confirms that only pure data (URLs, timestamps, lifecycle enums) are stored. The hypothesized failure mode (`navigationContext` storing a raw `Page` object) is successfully mitigated.

### Finding 3
**Severity**: None (Validation Success)
**Category**: Ephemeral Execution Handling
**Location**: `ActionSimulator.mjs`, `LocatorResolver.mjs`
**Description**: Playwright objects generated during execution (`LocatorResolver.resolve` yielding a Playwright `Locator`) are correctly managed ephemerally. They are passed back to the `ActionSimulator`, executed against, and immediately dropped when the local function scope exits, allowing for safe garbage collection.

## Answers to Blueprint Questions
1. **Does any DTO, State, or Context object store a framework-specific class?** Yes. The `BrowserRegistry` (legacy coordination layer) directly stores `Page`, `BrowserContext`, and `Browser` objects inside its state map.
2. **Are Playwright objects strictly passed by reference through function arguments?** No. While the newer `BrowserStateRegistry` handles this correctly, the `BrowserRegistry` explicitly caches these objects globally.

## Recommendations
1. **Remove `BrowserRegistry.mjs`**: Complete the migration away from `BrowserRegistry.mjs` by removing it entirely. 
2. **Decouple SessionManager**: Refactor `SessionManager.mjs` (which currently pulls the `page` from the registry) to receive the `page` as an ephemeral argument injected by the orchestration layer at execution time, or redesign it to issue `Command` payloads rather than executing Playwright scripts directly.

## Conclusion
While the newly built state models (`BrowserStateModel`) correctly restrict themselves to pure data, the system suffers from severe framework lock-in and memory leak risks due to the persistent caching of raw Playwright objects in the legacy `BrowserRegistry.mjs`.
