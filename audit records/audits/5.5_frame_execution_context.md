# Audit

## Domain
5.5 Frame Execution Context Isolation Audit

## Scope
- `ActionDispatcher.mjs` (Injection and Binding mechanisms)
- `FramePathBuilder.mjs`

## Architectural Intent
The automation engine must seamlessly record interactions inside nested and cross-origin iframes (e.g., third-party payment gateways like Stripe, or ad networks) without violating the browser's Same-Origin Policy (SOP). The slave must subsequently receive a deterministic path to route the playback commands into the exact same iframe.

## Findings

### Finding 1
**Severity**: None (Architectural Validation / Strength)
**Category**: Cross-Origin Penetration via CDP
**Location**: `ActionDispatcher.mjs` (Lines 36, 481)
**Description**: The blueprint asks: "How does the script inject itself into cross-origin iframes?"
**Analysis**: The architecture utilizes Playwright's `addInitScript` and `exposeBinding` APIs. Because these APIs operate at the Chrome DevTools Protocol (CDP) level rather than the DOM level, the injection occurs in every frame's V8 context simultaneously, completely bypassing DOM CORS restrictions. When an interaction occurs inside a cross-origin iframe (like a Stripe checkout), the injected `LocatorIntelligenceEngine` executes natively within that isolated context and transmits the payload back to Node.js via the `exposeBinding` side-channel, entirely avoiding `window.postMessage` and preventing any `SecurityError: Blocked a frame with origin X` exceptions.
**Conclusion**: This is a highly resilient architectural choice.

### Finding 2
**Severity**: Medium
**Category**: Non-Deterministic Frame Resolution
**Location**: `FramePathBuilder.mjs` (Lines 23-27)
**Description**: The blueprint asks: "Does it correctly identify the frame index/name for the slave to replicate?"
**Evidence**:
- Code E1: 
```javascript
path.unshift({
    isMainFrame: false,
    name: current.name(),
    url: current.url()
});
```
- Static Analysis E4: Playwright's `frame.name()` relies on the iframe's `name` or `id` HTML attribute. However, many third-party embeds (e.g., identical ad banners, tracking pixels, or dynamically generated widgets) lack a `name` attribute, resulting in an empty string `""`. If a page contains three identical unnamed iframes pointing to `https://third-party.com/widget`, the `FramePathBuilder` will record identical metadata (`name: "", url: "https://third-party.com/widget"`) for all three. When the slave attempts to replay the action, it lacks a deterministic identifier (such as an index) to differentiate between them, leading to flaky or incorrect execution.
**Impact**: Recording and playback will become non-deterministic on pages with multiple identical unnamed iframes.
**Root Cause**: Relying exclusively on metadata (`name`, `url`) rather than structural hierarchy (index or parent DOM locator).
**Recommendation**: Enhance `FramePathBuilder` to include a structural index. While Playwright's `Frame` object doesn't natively expose its DOM index directly, you can query the parent frame's DOM via CDP to find the index of the child frame's `<iframe>` element among its siblings. Alternatively, execute a DOM locator strategy *for the iframe element itself* within the parent context and append it to the path array.

## Overall Assessment
The interaction aggregator successfully circumvents cross-origin security restrictions through excellent utilization of CDP bindings. However, the resulting frame path metadata is insufficiently deterministic for complex, multi-frame environments.

## Recommendations
1. Modify `FramePathBuilder` to inject a script into the parent frame to calculate the exact CSS/XPath locator of the `<iframe>` element itself, or at minimum, an index (e.g., `frameIndex: 2`).

## Conclusion
The cross-origin injection is architecturally sound and immune to DOMExceptions. The only vulnerability is the non-deterministic resolution of identical sibling frames during playback.
