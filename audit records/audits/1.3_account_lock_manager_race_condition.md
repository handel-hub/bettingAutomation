# Audit

## Domain
1.3 Account Lock Manager Race Condition Audit

## Scope
- `src/browser/coordination/AccountLockManager.mjs`
- `src/browser/AutomationController.mjs` (Routing logic)
- `src/browser/execution/WorkflowEngine.mjs`
- `src/browser/workflows/CashoutWorkflow.mjs`

## Architectural Intent
The `AccountLockManager` is designed to enforce strict mutual exclusion (mutex) for automated workflows, ensuring that no two browsers attempt to perform business-critical operations (like Cashout) on the same user account concurrently.

## Files Examined
- `AccountLockManager.mjs`
- `AutomationController.mjs`
- `WorkflowEngine.mjs`
- `CashoutWorkflow.mjs`

## Execution Model
1. The `CommandRouter` in `AutomationController.mjs` receives a Workflow command.
2. It filters target browsers by checking `!this.lockManager.isLocked(b.username)`.
3. It passes the surviving targets to `WorkflowEngine.execute()`.
4. `WorkflowEngine.execute()` iterates over the targets, asynchronously calling `workflow.execute()`.
5. Inside `CashoutWorkflow.execute()`, `lockManager.acquireLock(username)` is called.
6. The workflow executes the business logic, occasionally calling `refreshLock()` within its main loop.
7. A `try/finally` block guarantees `releaseLock(username)` is called upon completion or failure.

## Findings

### Finding 1
**Severity**: Critical
**Category**: Concurrency / Race Condition
**Location**: `AutomationController.mjs` & `CashoutWorkflow.mjs`
**Description**: The system suffers from a classic Check-Then-Act (TOCTOU) race condition. `AutomationController` verifies that an account is not locked, but it does not actually acquire the lock. The lock is only acquired asynchronously later inside `CashoutWorkflow`. If two workflows targeting the same account are dispatched in rapid succession, both will evaluate `isLocked() === false` and proceed to execute concurrently.
**Evidence**:
- Code E1: `AutomationController.mjs` filters targets using `if (b.username && this.lockManager.isLocked(b.username)) { ... return false; }` but does not reserve the lock.
- Code E1: `CashoutWorkflow.mjs` acquires the lock (`lockManager.acquireLock`) only after the asynchronous routing boundary.
- Static Analysis E4: `AccountLockManager.acquireLock()` blindly overwrites existing locks (`this.locks.set(...)`) instead of throwing an error or returning `false` if the lock is already held. Thus, multiple concurrent acquisitions succeed silently.
- Execution Trace E2: 
  1. `t=0`: CommandRouter receives Workflow Command A targeting "user1".
  2. `t=1`: `AutomationController` checks `isLocked("user1")`. Returns `false`.
  3. `t=2`: `AutomationController` passes `slave_1` to `WorkflowEngine.execute()`, which launches async execution. Node event loop continues.
  4. `t=3`: CommandRouter receives Workflow Command B targeting "user1".
  5. `t=4`: `AutomationController` checks `isLocked("user1")`. Since `slave_1` hasn't acquired the lock yet (pending async callback), it returns `false`.
  6. `t=5`: `AutomationController` passes `slave_2` to `WorkflowEngine.execute()`.
  7. `t=6`: `slave_1` invokes `CashoutWorkflow.execute` and calls `acquireLock("user1")`.
  8. `t=7`: `slave_2` invokes `CashoutWorkflow.execute` and calls `acquireLock("user1")`. Both execute concurrently.
**Impact**: Business rule violation. Two browsers can simultaneously perform actions on the same account (e.g., cashing out the same ticket), resulting in server-side rejection, account flagging, or duplicate logic errors.
**Root Cause**: Decoupled lock verification and lock acquisition across an asynchronous event loop boundary.
**Recommendation**: Lock acquisition must be atomic. Implement a `tryAcquireLock(username)` method in `AccountLockManager` that returns `true` if successfully acquired and `false` if already locked. `AutomationController` should call `tryAcquireLock` during the filtering phase.
**Confidence**: Very High

### Finding 2
**Severity**: High
**Category**: Concurrency / State Corruption
**Location**: `CashoutWorkflow.mjs`
**Description**: Because `acquireLock` lacks mutual exclusion and allows multiple workflows to enter the critical section (Finding 1), the `finally` block's `releaseLock()` becomes highly destructive. When the first concurrent workflow completes, it deletes the lock for the account, leaving the second workflow still running completely unprotected.
**Evidence**:
- Code E1: `finally { if (lockManager) lockManager.releaseLock(accountUsername); }` unconditionally deletes the lock without verifying ownership.
- Static Analysis E4: `AccountLockManager.releaseLock(username)` maps solely to `this.locks.delete(username)`. It does not compare the execution ID or reference count of the caller. Therefore, an overlapping execution that finishes first will completely strip the lock from the still-running instance.
**Impact**: If a third workflow is dispatched while the second workflow is still executing (but after the first has released the lock), it will successfully enter the critical section. This creates a cascading failure of mutual exclusion.
**Root Cause**: Unconditional release of a non-reentrant, blindly overwritable lock without tracking lock ownership.
**Recommendation**: Tie lock ownership to a unique execution ID (e.g., UUID or `browserId`). `releaseLock(username, executionId)` should only delete the lock if the current owner matches the caller.
**Confidence**: Very High

### Finding 3
**Severity**: Medium
**Category**: Reliability
**Location**: `AccountLockManager.mjs` & `CashoutWorkflow.mjs`
**Description**: `AccountLockManager` enforces a hard TTL of 30,000ms. While `CashoutWorkflow` refreshes this lock inside its loop, the initial phase (navigation, waiting for the dynamic loading mask, querying ticket arrays) involves Playwright `.waitFor()` timeouts totaling up to 11,000ms. If network conditions are poor or Playwright retries internally, the execution can exceed 30 seconds before reaching the `refreshLock()` call, causing the lock to auto-expire mid-execution.
**Evidence**:
- Code E1: `CashoutWorkflow.mjs` lines 19-38 contain multiple asynchronous network-bound waits before the `while` loop begins.
**Impact**: The lock will silently expire during a slow network scenario. The next workflow command will see `isLocked() === false` and begin executing concurrently, violating exclusivity.
**Root Cause**: A fixed TTL with manual, sparse refreshing rather than an active heartbeat or execution-bound lock lifecycle.
**Recommendation**: Implement an active heartbeat inside the `WorkflowEngine` that automatically calls `refreshLock` every 10 seconds for any running workflow, rather than requiring the workflow implementation to manually remember to refresh it.
**Confidence**: High

### Finding 4
**Severity**: High
**Category**: State Desynchronization
**Location**: `AutomationController.mjs` & `CashoutWorkflow.mjs`
**Description**: The `UNIQUE_ACCOUNTS_ONLY` execution mode is intended to prevent multiple slaves sharing the same account from running workflows simultaneously. It relies on checking if any slave sharing the account is currently marked `Busy` (`isAnyBusy`). However, the workflows never update the registry to mark the browser as `Busy` during execution.
**Evidence**:
- Code E1: `AutomationController.mjs` relies on `const isAnyBusy = browsersForAccount.some(s => s.state === 'Busy');` for its safety check.
- Static Analysis E4: A search across `WorkflowEngine.mjs` and `CashoutWorkflow.mjs` reveals zero invocations of `registry.updateState(id, 'Busy')`. Since the state remains `Ready` throughout the workflow execution, the `isAnyBusy` check always evaluates to false, rendering the `UNIQUE_ACCOUNTS_ONLY` safety mechanism completely inoperative against overlapping commands.
**Impact**: Broadens the race condition window. Successive `UNIQUE_ACCOUNTS_ONLY` commands will aggressively target the same account via multiple slaves.
**Root Cause**: Missing state lifecycle emission within the execution engine.
**Recommendation**: `WorkflowEngine` must explicitly transition the browser state to `Busy` before executing the workflow and restore it to `Ready` (or `Error`) upon completion.
**Confidence**: Very High

## Positive Observations
- The use of `finally` blocks in `CashoutWorkflow` guarantees that under normal, single-threaded execution, locks are reliably released even during unhandled exceptions or Playwright crashes.
- `UNIQUE_ACCOUNTS_ONLY` routing mode successfully mitigates duplicate targeting within the *same* command broadcast, narrowing the race condition window to multiple distinct commands.

## Risks
- **RISK-NEW-03**: Check-Then-Act race condition in account locking.
- **RISK-NEW-04**: Unconditional lock release violating mutual exclusion for overlapped executions.
- **RISK-NEW-05**: Lock expiration due to slow initial network bounds.
- **RISK-NEW-06**: WorkflowEngine fails to emit Busy state, breaking UNIQUE_ACCOUNTS_ONLY safety checks.

## Architectural Weaknesses
The `AccountLockManager` is treated as a passive data store rather than a thread-safe synchronizer. By separating "checking" from "acquiring", it provides the illusion of safety without actual concurrency protection.

## Production Readiness
Not ready for production. The failure to enforce atomic lock acquisition guarantees that under high-load or repeated triggering, multiple slave browsers will act on the same user account simultaneously.

## Overall Assessment
The locking mechanism is fundamentally flawed due to Check-Then-Act anti-patterns. While the intent is correct, the implementation fails under concurrent load, and its intended fallbacks (like state-based filtering) are broken by missing lifecycle events.

## Recommendations
1. Modify `AccountLockManager.acquireLock` to be atomic (returning a boolean success).
2. Shift lock acquisition to the routing layer (`AutomationController.mjs`) so that targets are only selected if they actively secure the lock.
3. Introduce an `ownerId` to the lock structure to prevent arbitrary `releaseLock` calls from destroying another execution's lock.
4. Implement an automatic lock-refresh heartbeat wrapper in `WorkflowEngine` instead of relying on manual `refreshLock` calls.
5. Update `WorkflowEngine` to transition browser state to `Busy` prior to workflow initialization.

## Conclusion
The `AccountLockManager` fails to provide the required mutual exclusion. Atomic acquisition, owner tracking, state consistency, and automated heartbeats must be implemented to secure workflow exclusivity.
