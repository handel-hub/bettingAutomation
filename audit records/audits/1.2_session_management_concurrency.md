# Audit

## Domain
1.2 Session Management Concurrency Audit

## Scope
- `SessionManager.mjs`

## Architectural Intent
The `SessionManager` is designed to be the single source of truth for loading, verifying, and persisting authentication cookies (`sessions/<username>.json`) across the distributed master-slave topology. It must safely handle concurrent accesses when multiple slaves assigned to the same account attempt to recover or boot simultaneously.

## Files Examined
- `src/browser/coordination/SessionManager.mjs`

## Findings

### Finding 1
**Severity**: Critical
**Category**: Simultaneous UI Login Race Condition
**Location**: `SessionManager.mjs` (Lines 89-115, `restoreOrLogin`)
**Description**: The `restoreOrLogin` method acts as the entry point for session state recovery. However, it lacks any Promise deduplication or in-memory mutex locking keyed by `username`. If five slaves assigned to the same account boot simultaneously and the session file is missing or expired, all five will concurrently fail `verifyLoggedIn()`. Consequently, all five will independently execute `this.login()` (the Playwright UI macro).
**Evidence**:
- Code E1: Lines 89-115 lack any tracking (e.g., `this.inFlightLogins = new Map()`) to consolidate simultaneous login requests.
- Runtime Behavior E3 (Projected): Five concurrent headless browsers will hit the target betting site simultaneously to execute the login macro. The target site's risk engine will detect five simultaneous authentications from the same account across five different proxy IPs, resulting in a permanent account lock/ban. Furthermore, all five will concurrently race to execute `saveSession()` upon success.
**Impact**: Catastrophic account banning by target site risk engines, massive unnecessary CPU overhead, and disk write races.
**Root Cause**: Lack of Promise memoization/deduplication for asynchronous authentication flows.
**Recommendation**: Implement a `this.loginLocks = new Map<string, Promise>()` lock in `restoreOrLogin`. If a login for `username` is already in-flight, subsequent callers must `await` the existing Promise rather than spawning a new Playwright macro.
**Confidence**: Absolute

### Finding 2
**Severity**: High
**Category**: TOCTOU File I/O Race Condition
**Location**: `SessionManager.mjs` (Lines 29-31, `loadSession`)
**Description**: The system suffers from a Time-of-Check to Time-of-Use (TOCTOU) vulnerability during session loading. It synchronously checks if the file exists (`fs.existsSync(sessionFile)`), and then yields the Node.js event loop by calling `await fsPromises.readFile(...)`. If another browser instance belonging to the same account synchronously triggers `saveSession()` (which uses `fs.writeFileSync`), the file will be overwritten exactly while the async read stream is buffering.
**Evidence**:
- Code E1: Line 29 (`fs.existsSync`) followed immediately by an `await` on Line 31 (`fsPromises.readFile`).
- Static Analysis E4: Because `saveSession` writes files synchronously, it can seamlessly interleave between the async read chunks of `loadSession`, resulting in `JSON.parse` attempting to parse a corrupted, partially overwritten buffer, crashing the orchestrator.
**Impact**: Random workflow crashes via unhandled `SyntaxError: Unexpected end of JSON input`.
**Root Cause**: Mixing synchronous checks with asynchronous file reads, combined with a lack of inter-process or intra-process file locking (e.g., `proper-lockfile`).
**Recommendation**: Utilize atomic file operations. Write to a `.tmp` file and use `fsPromises.rename()` to atomic swap the session file, ensuring `readFile` never encounters a partially written state.
**Confidence**: Very High

### Finding 3
**Severity**: High
**Category**: Event Loop Starvation via Synchronous I/O
**Location**: `SessionManager.mjs` (Line 59, `saveSession`)
**Description**: The `saveSession` method uses `fs.writeFileSync` to persist the encrypted cookies. In a high-concurrency Node.js environment heavily reliant on fast IPC and WebSocket event dispatching (e.g., `SynchronizationCoordinator`), blocking the main thread for disk I/O is a severe architectural anti-pattern.
**Evidence**:
- Code E1: Line 59 invokes `fs.writeFileSync(sessionFile, JSON.stringify(encrypted, null, 2), { mode: 0o600 });`.
- Static Analysis E4: A disk write of an encrypted payload blocks V8's thread. If a network event triggers 50 simultaneous session saves, the entire BAAOS event loop will freeze for several milliseconds, causing synchronization barriers to timeout and slave heartbeats to drop.
**Impact**: System-wide latency spikes and cascading synchronization failures.
**Root Cause**: Use of synchronous disk I/O in a fully asynchronous event-driven system.
**Recommendation**: Refactor `saveSession` to use `await fsPromises.writeFile()` combined with atomic `.tmp` renaming to prevent blocking the event loop.
**Confidence**: Absolute

## Positive Observations
- The migration path for legacy plaintext cookies is elegantly integrated, automatically upgrading and encrypting old sessions on load.
- Encrypting the `cookies` payload with a unique IV/authTag mitigates local state tampering.

## Overall Assessment
The `SessionManager` fails to implement any concurrency control mechanisms. It is highly vulnerable to concurrent execution spikes which will trigger anti-bot bans via simultaneous multi-IP logins, and suffers from basic Node.js I/O race conditions.

## Recommendations
1. Implement a Promise-based mutex map to deduplicate concurrent login macros per username.
2. Refactor all file operations to use asynchronous `fsPromises` and atomic file renames (`renameSync`/`rename`) to eliminate TOCTOU corruption.
3. Remove all `*Sync` I/O operations from the hot path.

## Conclusion
The Session Management layer is fundamentally thread-unsafe within the asynchronous Node.js model and requires immediate locking mechanisms.
