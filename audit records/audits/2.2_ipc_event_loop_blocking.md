# Audit

## Domain
2.2 IPC Event Loop Blocking Audit

## Scope
- `CommandRouter.mjs` (Routing mechanisms and Promise handling)
- `AutomationController.mjs` (Handler definitions)
- `ActionDispatcher.mjs` (Handler downstream execution)

## Architectural Intent
The `CommandRouter` acts as the central Nervous System of the BAAOS framework, distributing commands (Macros, Workflows, Recoveries) across the cluster. Because Node.js utilizes a single-threaded V8 event loop, the router and its handlers must absolutely avoid synchronous CPU-bound operations or synchronous file I/O. Any blocking operation will freeze the IPC channel, causing Playwright WebSocket timeouts and desynchronizing the `SynchronizationCoordinator`'s strict timelines.

## Files Examined
- `src/browser/CommandRouter.mjs`
- `src/browser/AutomationController.mjs`
- `src/browser/execution/ActionDispatcher.mjs`

## Findings

### Finding 1
**Severity**: High
**Category**: Event Loop Starvation via Synchronous Handler I/O
**Location**: `ActionDispatcher.mjs` (Lines 143-152, `injectMasterListeners`)
**Description**: When the `CommandRouter` routes a `Recovery: MASTER_HEALED` command, the registered handler invokes `await this.actionDispatcher.injectMasterListeners(master.page)`. Inside `injectMasterListeners`, the code executes a synchronous `for` loop that performs 32 consecutive `fs.readFileSync` operations (to bundle the Locator Intelligence Pipeline files) and 1 `fs.writeFileSync` operation. Because these operations occur synchronously *before* the first `await` boundary, the Node.js event loop is completely frozen.
**Evidence**:
- Code E1: `AutomationController.mjs` Line 197 invokes `injectMasterListeners` inside the router handler. `ActionDispatcher.mjs` Line 146 invokes `fs.readFileSync` synchronously.
- Static Analysis E4: During this disk I/O blockade, `CommandRouter` is physically incapable of routing subsequent commands. Slave heartbeats, telemetry metrics, and barrier synchronization events will queue up and potentially timeout, leading to systemic desynchronization.
**Impact**: Intermittent cluster-wide latency spikes and Playwright CDP timeouts during Master recovery events.
**Root Cause**: Performing file bundling/compilation synchronously at runtime instead of caching it at application startup or using asynchronous streams.
**Recommendation**: Move the `pipelineFiles` concatenation logic into the `ActionDispatcher` constructor (which runs at startup) and cache the resulting `locatorIntelligenceCode` string in memory.

### Finding 2
**Severity**: Medium
**Category**: Router Execution Coupling Anti-Pattern
**Location**: `CommandRouter.mjs` (Lines 43-51, `route`)
**Description**: The `CommandRouter.route()` function maps over all registered handlers and explicitly awaits their completion via `await Promise.allSettled(promises)`. While this does not block the V8 thread (handlers execute concurrently as dangling Promises), it logically couples the router's dispatch phase to the payload's total execution phase.
**Evidence**:
- Code E1: Line 51 uses `await Promise.allSettled(promises)`. A handler like `Workflow` or `Execution` can take several minutes to run a complex betting sequence.
- Static Analysis E4: If any future system component, telemetry tracker, or API attempts to `await commandRouter.route(cmd)` to verify *dispatch success*, it will inadvertently block itself for the entire lifecycle of the 10-minute macro.
**Impact**: Future maintainability risks and accidental application deadlocks if the router is treated as a standard async dispatch bus.
**Root Cause**: Misunderstanding the architectural boundary between "routing an event" and "completing an execution lifecycle".
**Recommendation**: The router should simply execute `promises` in a fire-and-forget manner and return immediately, or return an execution ID that can be tracked via a separate Execution Registry.

## Positive Observations
- The core routing logic elegantly maps exact command types and wildcard (`*`) handlers, allowing flexible Event-Bus style publish/subscribe models.
- Handlers in `CommandRouter` are safely wrapped in individual `try/catch` blocks, preventing one failed handler from tearing down sibling handlers.
- The majority of file reading (e.g. `MacroEngine`, `SessionManager` macro loading) correctly uses asynchronous `fsPromises`.

## Overall Assessment
The IPC routing layer is fundamentally asynchronous, but it is compromised by a severely blocking downstream dependency (`injectMasterListeners`) that freezes the event loop during critical recovery operations.

## Recommendations
1. Refactor `ActionDispatcher` to statically bundle the Locator Intelligence Pipeline at boot time.
2. Remove `await Promise.allSettled` from `CommandRouter.route` to decouple dispatch from execution, aligning it with standard EventBus patterns.

## Conclusion
Resolving the synchronous disk I/O in the `MASTER_HEALED` flow is critical to ensuring the framework does not suffer from random synchronization lag during long execution windows.
