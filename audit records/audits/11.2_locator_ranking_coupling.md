# Audit

## Domain
11.2 Locator Ranking Strategy Coupling Audit

## Scope
- `LocatorIntelligenceEngine.mjs` (Replaces `LocatorEngine.mjs` from Blueprint)
- `CandidateValidator.mjs`
- `RankingEngine.mjs` & `RankingRules/`
- `LocatorResolver.mjs`

## Architectural Intent
The Locator Intelligence layer must remain a pure calculation engine. It should evaluate the fitness of candidate selectors via static string analysis, structural heuristics, and node metadata without executing DOM queries or coupling itself to the browser automation framework (e.g., calling `page.$`). All actual DOM execution and locator testing must be deferred to the framework orchestrator (`LocatorResolver.mjs`).

## Findings

### Finding 1
**Severity**: None (Validation Success)
**Category**: Zero Framework Coupling
**Location**: `RankingEngine.mjs`, `LocatorResolver.mjs`
**Description**: The hypothesized failure mode—where strategies or engines directly invoke `page.$`—is successfully mitigated.
**Evidence**:
- Static Analysis E4: The `RankingEngine` and its `RankingRules` operate purely mathematically based on pre-extracted heuristic metadata. No reference to `page`, Playwright, or Puppeteer exists anywhere in the `locatorIntelligence` directory.
- Code E1 (`LocatorResolver.mjs` Line 78): The framework execution is correctly deferred to the orchestrator. `LocatorResolver.mjs` receives the pre-ranked candidates and uses `page.locator(ctx.candidate.locator)` to safely test them against the live Playwright context.

### Finding 2
**Severity**: High
**Category**: Architectural Violation (Direct DOM Execution)
**Location**: `CandidateValidator.mjs`
**Description**: The codebase violates the core architectural rule stated in the blueprint: *"The Locator Engine should not assume it is running inside the browser, nor should it contain direct DOM access (`document.querySelector`)."*
**Evidence**:
- Code E1 (`CandidateValidator.mjs` Line 28): 
```javascript
const matches = document.querySelectorAll(candidate.locator);
matchCount = matches.length;
```
- Code E2 (`StructuralStrategy.mjs` Line 7):
```javascript
while (current && current !== document) {
```
- Static Analysis E4: The Intelligence Engine is currently designed to run injected inside the browser sandbox (via `ActionDispatcher`). During the `CandidateValidator` pipeline step, it directly executes `document.querySelectorAll()` to determine uniqueness (`status = 'UNIQUE' | 'AMBIGUOUS' | 'MISSING'`). This completely violates the separation of concerns. The intelligence engine is actively executing DOM searches to pre-validate locators, rather than restricting itself to heuristic metadata and leaving validation entirely to the orchestrator (`LocatorResolver`).

**Answers to Blueprint Questions:**
1. **Does the ranking engine invoke browser execution commands to test locators?** It does not invoke *framework* execution commands (e.g., `page.$`), but it DOES invoke *native* browser execution commands (`document.querySelectorAll`) inside the `CandidateValidator` step to test locator fitness.
2. **Is DOM manipulation strictly isolated to the Action Phase?** No. DOM traversal and native DOM querying are occurring inside the Locator Intelligence generation pipeline.

## Overall Assessment
The intelligence logic correctly avoids importing or utilizing Playwright commands. However, the system fails the deeper architectural mandate. By running the Intelligence Engine inside the browser context and actively querying the live DOM (`document.querySelectorAll`) to validate candidates, it creates a tight coupling to the browser environment and steals responsibilities that belong strictly to the `LocatorResolver`.

## Recommendations
1. **Relocate Validation**: Remove `CandidateValidator.mjs` from the in-browser `LocatorIntelligenceEngine` pipeline. 
2. **Defer to Orchestrator**: Shift uniqueness validation into `LocatorResolver.mjs`, utilizing Playwright's `locator.count()` (which already exists natively) rather than manual `document.querySelectorAll` loops.
3. **Pure String Heuristics**: Ensure the injected engine relies strictly on extracting attributes and paths, outputting raw unvalidated candidates that are ranked purely on static heuristics.

## Conclusion
The framework decoupling is successful, but the environmental decoupling failed. The Intelligence Engine improperly performs live DOM execution.
