# Audit

## Domain
4.2 Continuous Command Coalescing Math Audit

## Scope
- `ExecutionScheduler.mjs`
- `Command.mjs`

## Architectural Intent
The `ExecutionScheduler` manages four distinct priority queues. For high-frequency, low-impact events like `Continuous` (hover) and `Aggregated` (scroll), the scheduler coalesces rapid bursts of telemetry to prevent Event Loop starvation. A strict 1500ms TTL drops stale commands if the queue becomes terminally blocked.

## Findings

### Finding 1
**Severity**: High
**Category**: Stale Timestamp Inheritance
**Location**: `ExecutionScheduler.mjs` (Lines 43-71)
**Description**: When `SCROLL` or `wheel` events arrive rapidly, they are coalesced into the `Aggregated` queue. The mathematical integration correctly sums `deltaX` and `deltaY` into the `existing` (oldest) command in the queue. However, the scheduler fails to slide the `enqueueTime` forward. If a user performs a sustained scroll gesture lasting longer than 1.5 seconds, the accumulated command will trigger the 1500ms TTL expiration upon dequeue and be completely destroyed. 
**Evidence**:
- Code E1: In `SchedulingPolicy.apply`, `existing.command` is reconstructed with summed deltas, but `existing.enqueueTime` is never updated to match the newest `entry.enqueueTime`.
- Static Analysis E4: In `dequeueNext`, the TTL check `if (now - entry.enqueueTime > 1500)` will evaluate to true for a 3000ms continuous scroll, instantly dropping the massive coalesced scroll payload. 
**Impact**: Long, sustained scrolling gestures are systematically dropped by the queue, causing slave viewports to completely desync from the master's physical position.
**Root Cause**: Coalescing merges the payload but preserves the oldest lifecycle timestamps instead of inheriting the newest.
**Recommendation**: When successfully coalescing a payload in `SchedulingPolicy.apply`, explicitly update `existing.enqueueTime = entry.enqueueTime` so the TTL window slides forward with active user interaction.

### Finding 2
**Severity**: None (Positive Observation)
**Category**: Timezone Safety
**Location**: `Command.mjs` / `ActionDispatcher.mjs`
**Description**: The audit blueprint questioned whether timestamps are safely generated without timezone mismatches. All capture events are generated in the context of the master browser which resides on the same physical host as the node process. Timestamps rely exclusively on `Date.now()` (timezone-agnostic UTC epoch milliseconds) and `new Date().toISOString()`.
**Evidence**:
- Code E1: `Command.mjs` uses `Date.now()` for `captureTime` and `creationTime`.
**Impact**: The telemetry timestamps are mathematically immune to timezone drift.

## Overall Assessment
The coalescing mathematics for coordinate deltas are correct, but the lifecycle math handling TTL expiration is critically flawed for `Aggregated` queues. `Continuous` queues (like hover) correctly overwrite pending entries and slide their timestamps forward, rendering them immune to this bug.

## Recommendations
1. In `ExecutionScheduler.mjs`, add `existing.enqueueTime = entry.enqueueTime;` when a scroll is coalesced.
2. In the coalesced `new Command()` constructor, pass `captureTime: entry.command.captureTime` instead of `existing.command.captureTime` to ensure the command logically reflects the latest moment of the scroll interaction.

## Conclusion
Fixing the sliding window inheritance for coalesced scrolls will prevent massive bursts of inertia from being spontaneously deleted by the safety TTL.
