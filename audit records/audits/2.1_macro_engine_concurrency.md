# Audit

## Domain
2.1 Macro Engine Concurrency Audit

## Scope
- `src/browser/execution/MacroEngine.mjs`
- `src/browser/execution/ExecutionScheduler.mjs`

## Architectural Intent
The `MacroEngine` is responsible for parsing predefined sequences of actions and safely executing them across multiple browsers. These sequences must be atomic (not interleaved with competing actions) and logically sound.

## Files Examined
- `MacroEngine.mjs`
- `ExecutionScheduler.mjs`

## Execution Model
1. `MacroEngine.loadSequence()` reads a JSON file and parses it into `Command` objects.
2. `MacroEngine.validate()` iterates through the sequence and checks if all referenced DOM selectors exist and are visible on the target browsers.
3. `MacroEngine.execute()` iterates through the `Command` sequence for each browser. It enqueues one command at a time to the `ExecutionScheduler` and calls `await this.scheduler.waitForIdle(b.id)` before proceeding to the next command.
4. `ExecutionScheduler._drain()` pulls commands from queues and executes them via `ActionSimulator`. It catches any execution exceptions and suppresses them.

## Findings

### Finding 1
**Severity**: Critical
**Category**: Error Handling / State Corruption
**Location**: `MacroEngine.execute()` & `ExecutionScheduler._drain()`
**Description**: The `MacroEngine` blindly iterates through sequence steps without receiving success/failure feedback from the `ExecutionScheduler`. If an early macro step (e.g., clicking a navigation button) fails, the `ExecutionScheduler` swallows the error, releases the drain lock, and allows the `MacroEngine` to enqueue the next step.
**Evidence**:
- Code E1: `ExecutionScheduler.mjs` line 289 `catch(e) { logger.error(...); }` suppresses the exception.
- Code E1: `MacroEngine.mjs` line 74 calls `await this.scheduler.waitForIdle(b.id)` without checking the result of the operation.
- Execution Trace E2: 
  1. `t=0`: `MacroEngine` enqueues Step 1 (Click `#login`). `_drain` acquires lock.
  2. `t=1`: `_drain` invokes `simulator.execute`. Playwright throws "Timeout: selector #login not found".
  3. `t=2`: `_drain` catches exception, logs it, and continues. Queue is empty. `_drain` completes and releases `drainLocks`.
  4. `t=3`: `waitForIdle` resolves in `MacroEngine`.
  5. `t=4`: `MacroEngine` blindly enqueues Step 2 (Type "password" into `#pwd`).
  6. `t=5`: `_drain` attempts Step 2 on an unauthenticated page, injecting sensitive data into potentially malicious or unrelated DOM states.
**Impact**: Macros will continue executing against an invalid or completely wrong DOM state. This leads to unpredictable side effects, interacting with wrong elements, or causing orchestrated chaos across slave browsers.
**Root Cause**: Lack of a bidirectional feedback loop between the asynchronous scheduler and the sequence generator.
**Recommendation**: `ExecutionScheduler` must expose a mechanism (e.g., returning a Promise from `enqueue` or emitting a sequence-failure event) to inform the `MacroEngine` of command success. `MacroEngine` must abort the sequence loop immediately upon the first failure.
**Confidence**: Very High

### Finding 2
**Severity**: Critical
**Category**: Logic Failure
**Location**: `MacroEngine.validate()`
**Description**: The `validate()` method iterates through every command in the sequence and immediately evaluates the DOM using `locator.count()` and `locator.isVisible()`. This static validation fundamentally misunderstands web application state flow. Many elements in a sequence (e.g., a modal dialogue, a dropdown menu) do not exist or are not visible until earlier steps in the sequence are executed.
**Evidence**:
- Code E1: `MacroEngine.mjs` lines 43-53 contain a tight `for` loop that evaluates every selector against the *current* page state sequentially.
- Execution Trace E2: A macro is loaded: Step 1: Click "Menu" (`#menu`), Step 2: Click "Logout" (`#logout`). `validate()` checks `#menu` (exists) then immediately checks `#logout` (hidden because the menu hasn't been clicked yet). Validation throws "Selector present but not visible: #logout" and aborts.
**Impact**: Any dynamic macro that traverses states or reveals new elements will universally fail validation and never execute. 
**Root Cause**: Attempting static DOM validation on sequential, state-mutating steps.
**Recommendation**: Remove the static `validate()` pre-flight check entirely. Rely instead on Playwright's native auto-waiting mechanisms and ensure Finding 1 is fixed so that the macro aborts gracefully at runtime if a selector fails to appear.
**Confidence**: Very High

### Finding 3
**Severity**: High
**Category**: Concurrency / Atomicity
**Location**: `MacroEngine.execute()`
**Description**: `MacroEngine.execute()` submits commands to the scheduler individually and yields the event loop while waiting for `waitForIdle`. This design explicitly surrenders exclusivity between every macro step. If a broadcast command (e.g., an interactive click from the Master browser) arrives during this idle gap, the `ExecutionScheduler` will process it in the middle of the macro sequence.
**Evidence**:
- Code E1: `await this.scheduler.waitForIdle(b.id)` awaits a polling loop.
- Static Analysis E4: During the `await`, the `drainLocks` set is empty. If `AutomationController` invokes `scheduler.enqueue` in response to an asynchronous Event Bus message, the scheduler will instantly acquire the lock and execute the injected command before the `MacroEngine` resumes to enqueue the next step.
**Impact**: Interleaved commands break the atomicity of macro sequences, polluting the DOM state and causing race conditions that derail the orchestration.
**Root Cause**: Step-by-step enqueuing into a shared, priority-based execution queue.
**Recommendation**: The `MacroEngine` should enqueue the *entire sequence* as a single atomic batch, or the `ExecutionScheduler` must support an `ExecutionLock` that pauses non-critical inbound routing while a macro sequence is actively holding the lock.
**Confidence**: Very High

### Finding 4
**Severity**: High
**Category**: State Corruption / Queue Consistency
**Location**: `MacroEngine.execute()` & `ExecutionScheduler.handleNavigation()`
**Description**: If an interleaved command during a macro sequence is a `Navigation` command, the `ExecutionScheduler` will explicitly flush its discrete event queues. However, because the `MacroEngine` loops asynchronously outside of the scheduler, it is entirely unaware of this flush. Once the navigation starts, the engine will blissfully enqueue the *next* step of the sequence onto the new webpage.
**Evidence**:
- Code E1: `ExecutionScheduler.handleNavigation()` flushes `this.buckets.Discrete`.
- Static Analysis E4: A `Navigation` event occurs between Macro Step 2 and Step 3. The scheduler queue is flushed. The browser navigates. `MacroEngine` loop awakens and calls `this.scheduler.enqueue(Step 3)`. The scheduler successfully processes Step 3 on the target URL (if it loads fast enough), executing sequence actions in a foreign context.
**Impact**: Severe state corruption. Workflows meant for one URL will execute their keystrokes or clicks on an entirely different URL post-navigation.
**Root Cause**: Subsystem decoupling without lifecycle synchronization.
**Recommendation**: `ExecutionScheduler` must emit a `QueueFlushed` or `NavigationStarted` event that `MacroEngine` listens to, instantly aborting any currently executing `for` loop sequences for that browser.
**Confidence**: Very High

## Positive Observations
- Error boundaries inside the `ExecutionScheduler` prevent node crashes (despite suppressing the errors for the MacroEngine).
- Parallel execution of macros across `targetBrowsers` using `Promise.allSettled` is implemented correctly to maximize throughput.

## Risks
- **RISK-NEW-07**: MacroEngine continues executing sequences after step failures, corrupting states.
- **RISK-NEW-08**: Static sequence validation mathematically guarantees failure for dynamic UI flows.
- **RISK-NEW-09**: Loss of sequence atomicity due to scheduler yielding between steps.
- **RISK-NEW-10**: Cross-subsystem queue flushing causing partial macro execution on incorrect URLs.

## Architectural Weaknesses
The architecture incorrectly treats a Macro sequence as a series of independent commands rather than an atomic transaction. Furthermore, it assumes the DOM is statically verifiable before execution, ignoring the reality of stateful SPAs.

## Production Readiness
Not ready. The lack of error propagation, sequence atomicity, and cross-subsystem synchronization guarantees that macros will fail destructively under real-world orchestration loads.

## Overall Assessment
The `MacroEngine` is functionally broken for dynamic sequences and fundamentally unsafe for state mutation due to missing failure aborts and atomicity guarantees.

## Recommendations
1. Implement error propagation from `ExecutionScheduler` to abort macros mid-sequence upon failure.
2. Delete `MacroEngine.validate()`.
3. Introduce an `acquireSequenceLock(browserId)` mechanism in the scheduler to buffer or drop incoming broadcast events while a macro is executing.
4. Integrate a `NavigationStarted` abort signal to kill active macro loops immediately upon queue flushes.

## Conclusion
The `MacroEngine` fails its core mandate to safely coordinate execution sequences. Overhauls to validation logic, error handling boundaries, and queue atomicity are mandatory.
