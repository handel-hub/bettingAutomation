# Audit

## Domain
3.3 Barrier Timeout & Starvation Audit

## Scope
- `src/browser/synchronization/SynchronizationBarrier.mjs`
- `src/browser/synchronization/SynchronizationManager.mjs`
- `src/browser/synchronization/providers/DOMCapabilityProvider.mjs`

## Architectural Intent
The `SynchronizationBarrier` acts as a stateless execution gate. It must evaluate capabilities provided by the `SynchronizationManager` and either resolve (allowing execution) or reject/timeout (aborting execution). Crucially, the barrier must honor the configured `deadline` timeout to prevent event loop exhaustion and zombie contexts.

## Files Examined
- `SynchronizationBarrier.mjs`
- `SynchronizationManager.mjs`
- `DOMCapabilityProvider.mjs`
- `NavigationWaitStrategy.mjs`

## Execution Model
1. The `ExecutionScheduler` invokes `SynchronizationBarrier.wait(syncContext)`.
2. The Barrier enters a `while(true)` evaluation loop.
3. It delegates to `SynchronizationManager.awaitCapabilities()`, which executes `Promise.allSettled(providers)`.
4. The providers (e.g., `DOMCapabilityProvider`) execute their respective `waitFor` logic.
5. Once all providers settle, the barrier checks if the `Date.now() >= deadline`. If so, it returns `TIMEOUT`.

## Findings

### Finding 1
**Severity**: Critical
**Category**: Event Loop Starvation / Timeout Bypass
**Location**: `SynchronizationBarrier.mjs` & `SynchronizationManager.mjs`
**Description**: The barrier's global `deadline` timeout is logically misplaced. The timeout is only evaluated *after* `await SynchronizationManager.awaitCapabilities(...)` resolves. If any single `CapabilityProvider` hangs indefinitely (e.g., `DOMCapabilityProvider` waiting on a hung IPC `page.evaluate` call), the `awaitCapabilities` method blocks forever. The execution context never reaches the timeout evaluation.
**Evidence**:
- Code E1: `SynchronizationBarrier.mjs` lines 64-110 places the timeout `if (Date.now() >= deadline)` sequentially after the `awaitCapabilities` loop, rather than wrapping the `await` in a global `Promise.race([..., timeoutTimer])`.
- Execution Trace E2: 
  1. `t=0`: Barrier invoked with `deadline` of `t=5000`. `awaitCapabilities` is called.
  2. `t=1`: `DOMCapabilityProvider` injects an asynchronous `page.evaluate` script.
  3. `t=2`: The Playwright IPC socket silently drops. `page.evaluate` blocks indefinitely because it lacks an internal timeout.
  4. `t=5000`: The global deadline expires. However, because the main async task is yielded to `awaitCapabilities`, the loop is suspended.
  5. `t=Inf`: The `_drain` loop in `ExecutionScheduler` hangs forever.
**Impact**: Total event loop starvation for the affected slave browser. The execution queue will fill up indefinitely, memory will leak, and the slave becomes an unrecoverable zombie process that never times out.
**Root Cause**: Relying on child promises to resolve before evaluating the parent timeout.
**Recommendation**: Wrap the `awaitCapabilities` call inside a `Promise.race` with a strict `setTimeout` rejecting at the `deadline`. Alternatively, pass an `AbortSignal` down to all providers and invoke `.abort()` when the deadline triggers.
**Confidence**: Very High

### Finding 2
**Severity**: High
**Category**: Unbounded Memory Leak
**Location**: `SynchronizationManager.mjs`
**Description**: The `SynchronizationManager` manages state using an `activeSyncContexts` Map. It stores the heavy `syncContext` object (which holds references to the `page`, `browserContext`, and command payloads) before executing providers, expecting to delete it in a `finally` block. Because providers can hang (as proven in Finding 1), the `finally` block never executes.
**Evidence**:
- Code E1: `SynchronizationManager.mjs` line 51 calls `this.activeSyncContexts.set()`. Line 57 calls `this.activeSyncContexts.delete()` inside `finally`.
- Static Analysis E4: A suspended async execution halts the function scope permanently. The `Map` retains the reference to `syncContext` forever. Even if the slave browser process is externally killed, the Node.js V8 Garbage Collector cannot reclaim the Playwright objects because they are anchored in the active `Map`.
**Impact**: Massive memory leak. A few starved barriers will eventually cause an Out-Of-Memory (OOM) crash for the entire orchestration Node process.
**Root Cause**: Coupling long-lived state maps to the resolution of untrusted asynchronous provider promises.
**Recommendation**: Use a `WeakMap` keyed by `browserId` or `page` object instead of a standard `Map`. Furthermore, resolving Finding 1 will ensure the `finally` block always executes.
**Confidence**: Very High

## Positive Observations
- The `NavigationWaitStrategy` correctly manages its own `setTimeout` and cleans up its event listeners via `cleanup()`, ensuring it does not contribute to the memory leak if a timeout occurs internally.
- The use of `Promise.allSettled` instead of `Promise.all` correctly prevents a single rejected provider from instantly short-circuiting the telemetry gathering process.

## Risks
- **RISK-NEW-15**: Event loop starvation leading to permanent zombie slave contexts.
- **RISK-NEW-16**: Unbounded memory leaks in `activeSyncContexts` causing catastrophic Node.js OOM crashes.

## Architectural Weaknesses
The synchronization architecture incorrectly assumes that internal framework code (Capability Providers) is perfectly reliable. It lacks a global supervisor or `AbortController` hierarchy to aggressively terminate hung operations.

## Production Readiness
Not ready. The vulnerability to zombie starvation and memory leaks makes the orchestration platform fundamentally unstable under real-world network turbulence.

## Overall Assessment
The `SynchronizationBarrier` fails its primary objective: acting as a definitive execution gate with guaranteed timeouts. The architecture requires a shift towards aggressive timeout enforcement via `Promise.race` or `AbortSignal`.

## Recommendations
1. Implement a global `Promise.race` timeout at the `SynchronizationBarrier.wait()` level.
2. Refactor `activeSyncContexts` to use a `WeakMap`.
3. Add a mandatory `AbortSignal` to the `CapabilityProvider` interface so providers can proactively cancel pending Playwright operations when the barrier times out.

## Conclusion
The Barrier is incapable of enforcing its own timeouts. Structural changes to promise orchestration and memory management are mandatory.
